# memflux Architecture

## Overview

memflux is a distributed in-memory key-value cache with a single-leader, multi-follower replication model. All writes go to the leader, which synchronously replicates them to followers over gRPC. Any node can serve reads from its local store. Nodes are started as independent processes and discover each other through command-line flags.

---

## Storage

Every node, whether leader or follower, owns an identical `Store` struct. The store is an in-memory hash map protected by a `sync.RWMutex`. Each entry holds a value string and an optional expiry timestamp. If a TTL of zero is passed, the entry never expires.

Reads acquire a read lock, check whether the key exists, and then check whether it has expired before returning. Writes acquire a write lock and compute the expiry time as `now + ttl` before inserting.

A background goroutine called `sweep` runs on every node and ticks every second. On each tick it does two passes over the map: the first pass (read lock) collects all expired keys, and the second pass (write lock) re-checks each one and deletes it if it is still expired. The double-check guards against a key being updated between the two passes. Each deletion increments an eviction counter. The store also tracks hit and miss counts using atomic increments so that stats can be read without holding a lock.

---

## Node Roles

**Leader** — there is exactly one leader. It is the only node that accepts write requests from clients. When a write comes in, the leader applies it to its own store and then fans it out to all healthy followers via gRPC. The leader is started with `--role=leader` and a `--followers` flag listing the gRPC addresses of every follower.

**Follower** — there can be any number of followers. Each follower accepts read requests from clients and replication requests from the leader. Followers do not accept writes directly from clients; they receive all mutations through the replication channel. A follower is started with `--role=follower` and a `--leader` flag pointing at the leader's gRPC address.

---

## gRPC Replication Layer

All inter-node communication uses gRPC over plain TCP (no TLS). The proto service is called `Replication` and exposes two RPCs.

`Replicate` carries a single mutation from the leader to a follower. The request includes the operation (`set` or `delete`), the key, the value, the TTL in seconds, and a monotonically increasing sequence number generated by an atomic counter on the leader. The follower's handler applies the operation directly to its local store and returns a success/failure response. The sequence number is transmitted but not currently validated by the receiver; because gRPC uses HTTP/2 over TCP there is guaranteed in-order delivery on a single connection, so ordering is preserved without explicit enforcement.

`HeartBeat` is a liveness probe. It accepts a request carrying the caller's node ID and returns `{alive: true}`. This RPC is used in two places: the follower's heartbeat goroutine calls it on the leader to detect leader failure, and the leader's recovery goroutine calls it on unhealthy followers to detect when they come back.

Every node runs a gRPC server that registers the `Replication` service. The server listens on the port given by `--port` (default `8001`). On the leader, the gRPC port receives heartbeat probes from followers. On followers, it receives both replication calls and recovery probes from the leader.

---

## Leader Client & Replication Fan-out

When the leader starts, it creates a `LeaderClient` that holds a persistent gRPC connection to each follower. Each connection is wrapped in a `Follower` struct that also tracks a `healthy` boolean, protected by a mutex.

When a write is committed to the leader's store, `LeaderClient.Replicate` is called. It iterates over all followers, skips any marked unhealthy, and sends the `Replicate` RPC to each healthy one. Each send is attempted up to three times with a 500ms deadline per attempt. If all three attempts fail for a given follower, that follower is marked unhealthy and excluded from future replication until it recovers.

A goroutine called `recoverFollowers` runs in the background on the leader and ticks every 10 seconds. On each tick it probes every unhealthy follower with a `HeartBeat` RPC. If the probe succeeds, the follower is marked healthy again and replication resumes on the next write.

---

## HTTP API

Every node exposes a small HTTP server on `--http-port` (defaults to gRPC port + 1000, so a node on `8001` gets HTTP on `9001`). Three endpoints are available.

`GET /get?key=K` reads a key from the local store and returns the value as plain text, or a 404 if the key is missing or expired.

`POST /set` accepts a JSON body `{"key": "...", "value": "...", "ttl": N}` where `ttl` is in seconds. On the leader, the handler writes to the local store and then calls `lc.Replicate` to push the change to followers. On a follower, `lc` is nil so only the local store is written — but followers should not normally receive write traffic; this is a fallback.

`POST /delete` accepts a JSON body `{"key": "..."}` and behaves the same way as set: local delete followed by replication fan-out on the leader.

---

## Follower Heartbeat

Each follower runs a goroutine that dials the leader's gRPC address and sends a `HeartBeat` RPC every 5 seconds with a 1-second timeout. If the probe fails, the follower logs that the leader is unreachable. This is currently only for observability — the follower takes no automated action in response to a leader failure because there is no leader election implemented.

---

## Startup Order

When a node starts it parses flags, creates a store and starts the sweep goroutine, creates a `ReplicationServer` wrapping the store, starts the gRPC server in a goroutine, then branches on role. If it is a leader it creates the `LeaderClient` and starts the recovery goroutine. If it is a follower it starts the heartbeat goroutine. Finally it starts the HTTP server in a goroutine and blocks on an empty `select{}`.

---

## What Is Not Implemented

**Leader election.** If the leader goes down, followers continue serving reads from their local data but there is no mechanism to elect a new leader or redirect client writes.

**Write forwarding.** If a client sends a write to a follower's HTTP API, it only touches the follower's local store and is not replicated. Clients are expected to know which node is the leader.

**Read consistency.** Followers may serve slightly stale data in the window between a leader write and the replication completing. There is no read-your-writes or linearizability guarantee.

**Persistent storage.** All data lives in memory. A node restart loses all data on that node; the leader would need to replay all writes to a restarted follower, which is not implemented.
